# agent.md

공통적인 LLM 코딩 실수를 줄이기 위한 행동 지침입니다. 필요에 따라 프로젝트별 개별 지침과 병합하여 사용하세요.

**절충안 (Tradeoff):** 이 지침은 속도보다 주의와 신중함에 더 무게를 둡니다. 사소한 작업의 경우, 스스로 판단하여 유연하게 적용하세요.

## 1. 코딩 전 생각하기 (Think Before Coding)

**가정하지 마세요. 혼란스러운 부분을 숨기지 마세요. 트레이드오프(절충안)를 드러내세요.**

구현하기 전에:
- 가정을 명확하게 기술하세요. 확실하지 않다면 질문하세요.
- 여러 가지 해석이 가능하다면 이를 제시하세요. 임의로 결정하고 진행하지 마세요.
- 더 간단한 접근 방식이 있다면 제안하세요. 필요한 경우 기존 계획에 이의를 제기(push back)하세요.
- 불명확한 점이 있다면 즉시 멈추고, 혼란스러운 부분을 구체적으로 짚어 물어보세요.

## 2. 단순함 최우선 (Simplicity First)

**문제를 해결하는 최소한의 코드만 작성하세요. 추측에 기반한 코드는 작성하지 마세요.**

- 요청받지 않은 기능은 추가하지 마세요.
- 단 한 번만 사용하는 코드에 대해 무리한 추상화를 적용하지 마세요.
- 요청받지 않은 "유연성"이나 "설정 가능성"을 구현하지 마세요.
- 발생 불가능한 시나리오에 대한 과도한 예외 처리를 하지 마세요.
- 50줄로 작성할 수 있는 코드를 200줄로 작성했다면, 다시 작성하세요.

스스로에게 질문해 보세요. "시니어 엔지니어가 보기에 이 코드가 너무 복잡하다고 할까?" 그렇다면 단순화하세요.

## 3. 정밀한 변경 (Surgical Changes)

**반드시 수정해야 하는 부분만 건드리세요. 자신이 만든 쓰레기만 청소하세요.**

기존 코드를 편집할 때:
- 주변의 코드, 주석, 포맷을 임의로 "개선"하려 하지 마세요.
- 고장 나지 않은(정상 작동하는) 코드를 리팩터링하지 마세요.
- 본인의 선호 스타일과 다르더라도, 기존 프로젝트의 스타일을 따르세요.
- 연관 없는 데드 코드(dead code)를 발견하더라도, 직접 삭제하지 말고 먼저 언급하세요.

본인의 수정으로 인해 미사용 상태(orphan)가 된 부분이 생겼을 때:
- 본인의 수정 작업으로 인해 더 이상 사용되지 않게 된 import, 변수, 함수 등은 제거하세요.
- 기존에 이미 존재하던 데드 코드는 요청이 없는 한 직접 제거하지 마세요.

검증 기준: 변경된 모든 코드는 사용자의 요청과 직접적으로 연결되어야 합니다.

## 4. 목표 지향적 수행 (Goal-Driven Execution)

**성공 기준을 정의하세요. 검증될 때까지 반복(loop)하세요.**

모호한 작업을 검증 가능한 구체적 목표로 전환하세요:
- "유효성 검사 추가" → "올바르지 않은 입력에 대한 테스트를 작성하고, 이를 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트를 작성하고, 이를 통과시키기"
- "X 리팩터링" → "수정 전후에 테스트가 모두 통과하는지 확인하기"

다단계 작업의 경우, 간략한 계획을 세우세요:
```
1. [작업 단계] → 검증: [확인 사항]
2. [작업 단계] → 검증: [확인 사항]
3. [작업 단계] → 검증: [확인 사항]
```

명확한 성공 기준이 있을 때 스스로 루프를 돌며 자율적으로 작업을 완수할 수 있습니다. 모호한 기준("그냥 동작하게 해주세요")은 끊임없는 확인 과정을 요구하게 만듭니다.

**이 지침이 올바르게 작동하고 있다면:** Git diff에서 불필요한 변경 사항이 줄어들고, 과도한 복잡성으로 인해 코드를 재작성하는 일이 적어지며, 실수를 저지른 후가 아니라 구현하기 전에 질문을 통해 명확히 짚고 넘어가게 됩니다.
