# 일본어 스크립트의 한국어 번역 지침

## 목적과 적용 범위

이 문서는 이 저장소에서 `*_jpn.json`을 기반으로 같은 이름의
`*_kor.json`을 만들거나 수정할 때 적용한다. 번역문은 원본 바이너리의
동일한 주소 범위에 다시 기록되므로, 자연스러운 번역뿐 아니라 주소와
바이트 길이를 정확히 보존해야 한다.

참고 우선순위는 다음과 같다.

1. 같은 워크스페이스의 기존 `*_jpn.json`/`*_kor.json` 번역 쌍
2. 같은 파일과 인접 주소의 문맥
3. `custom_word.json`, `custom_char.json`
4. 저장소의 `name_db/` 및 게임별 용어 자료

같은 일본어 문장이 기존 번역 쌍에 있으면 먼저 그 번역을 재사용한다.
다만 화자, 조사, UI 길이 등 문맥이 다른 경우에는 문맥에 맞게 조정한다.

## 반드시 지켜야 할 불변 조건

1. `START=END` 형식의 주소 시작점과 항목 순서는 원본과 동일하게
   유지한다. 번역문이 들어가지 않을 때만 원본 바이너리에서 다음
   항목까지의 구간이 실제 `00`인지 확인한 뒤 끝 주소를 필요한 만큼
   확장하고, 양쪽 JSON에 같은 주소를 적용한다.
2. 주소 키의 순서도 원본 순서를 유지한다.
3. 각 한국어 값의 인코딩 바이트 길이는 반드시
   `int(END, 16) - int(START, 16) + 1`과 같아야 한다.
4. 문자열 안의 서식 인자, 제어 코드, 포인터 및 원시 바이트를 일반
   문장으로 오인해 번역하지 않는다.
5. 한국어 폰트 테이블이나 `custom_word.json`에 없는 문자를 사용하지
   않는다.
6. JSON은 UTF-8, 유효한 객체 형식으로 저장한다. 파일명은 원본의
   `_jpn.json`만 `_kor.json`으로 바꾼다.

`write_script_auto.py`의 일반 문자열 기록 부분은 주소의 끝을 기준으로
자동 차단하지 않는다. 길이가 초과되면 다음 데이터까지 덮어쓸 수
있으므로 기록기를 실행하기 전에 별도로 길이를 검증해야 한다.

## 스크립트 표기와 바이트 계산

| 표기 | 의미 | 길이 |
| --- | --- | ---: |
| 일반 문자 `가`, `_`, `.` 등 | 한국어 폰트 테이블의 2바이트 문자 | 문자당 2바이트 |
| `|X` | `X` 한 글자를 1바이트 테이블로 기록 | 1바이트 |
| `|_` | 1바이트 공백 | 1바이트 |
| `_` | 2바이트 공백 또는 길이 맞춤용 문자 | 2바이트 |
| `@` | 원래 바이너리 2바이트를 덮어쓰지 않고 건너뜀 | 2바이트 |
| `{키}` | `custom_word.json`의 바이트열 사용 | 해당 16진수 문자열 길이의 절반 |
| `0x:16진수#설명` | 문자 변환 없이 원시 바이트 기록 | 16진수 문자열 길이의 절반 |

`|`는 독립된 문자가 아니라 바로 다음 한 글자만 1바이트로 만드는
접두 표기다. 따라서 ASCII 서식 문자열은 다음과 같이 각 글자 앞에
`|`를 붙인다.

- `%s` → `|%|s`
- `%d` → `|%|d`
- `%4d` → `|%|4|d`
- `DUMMY` → `|D|U|M|M|Y`

일반 문장 안에 실제 ASCII 공백을 직접 넣지 않는다. 한국어 어절 사이는
기존 번역본처럼 주로 `|_`를 사용하고, 2바이트 여백이 필요할 때만 `_`를
사용한다.

화면에 표시되는 마침표·쉼표·물음표·느낌표는 `|.`, `|,`, `|?`, `|!`
같은 1바이트 표기가 아니라 `.`, `,`, `?`, `!`의 2바이트 글자로
기록한다. 파일명, `printf` 형식 문자열 등 모든 글자가 `|X`로 구성된
기능성 ASCII 문자열은 원래 바이트 배열을 보존한다.

예를 들어 `"가|_나|␀"`의 길이는 `2 + 1 + 2 + 1 = 6`바이트다.

## 제어 문자와 특수 데이터

`custom_char.json`을 기준으로 PC-98 예제에서 확인되는 주요 표기는
다음과 같다.

| 표기 | 기록 바이트 | 용도 |
| --- | ---: | --- |
| `|␀` / `␀` | `00` / `0000` | 1바이트 또는 2바이트 널·여백 |
| `|␁` | `01` | 게임 제어 코드 |
| `|␂` | `0A` | 대사 줄바꿈 |
| `|␊` | `1B` | 게임 제어 코드 |

- `%s`, `%d` 등의 서식 인자는 원문의 개수와 순서를 유지한다.
- `|␁`, `|␊` 같은 기능성 제어 코드는 의미를 확인하지 않고 삭제하거나
  다른 문자로 바꾸지 않는다.
- `|␂` 줄바꿈은 화면 폭에 맞춰 위치를 조정할 수 있지만, 대사창의
  줄 수와 표시 흐름을 고려해야 한다.
- 남는 공간은 문맥과 기존 파일의 관례에 따라 `_`, `|_`, `␀`, `|␀`
  등으로 채운다. 종료 코드가 필요한 위치의 널을 단순 공백으로
  교체하지 않는다.
- `0x:` 값에서는 `#` 앞의 16진수만 실제로 기록된다. 16진수의 글자 수는
  짝수여야 하며 주소 길이와 정확히 맞아야 한다. `#` 뒤 설명만 한글로
  정리할 수 있고, 바이트 값은 명확한 패치 근거가 없으면 변경하지 않는다.

## `custom_word.json` 사용

번역문이 주소 범위에 들어가지 않거나 고정된 용어·이름을 압축 코드로
써야 할 때 `{키}`를 사용한다.

```json
{
    "00919=00920": "{다이묘}정보"
}
```

- 중괄호 안에는 `custom_word.json`에 실제로 존재하는 키만 넣는다.
- 키에 `|`나 `_`가 포함되어 있어도 키 문자열 전체를 그대로 사용한다.
- 임의의 `{새키}`를 만들지 않는다. 새 키가 필요하면 충돌하지 않는
  바이트 코드와 폰트 구성을 별도로 검토한 뒤 `custom_word.json`을 함께
  수정해야 한다.
- `{키}`의 화면상 글자 수가 아니라 매핑된 16진수의 바이트 수로 길이를
  계산한다.
- 조사(`은`, `는`, `이`, `가`, `을`, `를`, `와`, `과`, `에`, `의`)의
  2바이트 압축 코드는 조사 뒤의 반각 공백까지 포함한다. 따라서
  `는|_`는 `{는}`으로 바꿀 수 있으며, `는|_|_`처럼 공백을 다시
  덧붙이지 않는다.

## 번역 원칙

1. 직역보다 게임 화면에서 자연스럽고 짧은 한국어를 우선한다.
2. 화자의 신분과 기존 말투를 유지한다. 같은 인물의 존대, 하오체,
   명령조가 대사마다 흔들리지 않게 한다.
3. 인명, 지명, 관직, 아이템명은 기존 번역 쌍과 데이터베이스의 표기를
   우선한다.
4. UI 명칭과 반복 문구는 같은 번역을 사용한다.
5. 서식 인자에 조사가 붙으면 실제 치환 결과를 고려한다. 필요하면
   `custom_word.json`의 조사 관련 키를 사용하되, 기존 사용례를 먼저
   확인한다.
6. 화면 폭을 고려해 `|␂` 위치를 잡고, 한 줄이 지나치게 길어지지 않게
   문장을 축약한다.
7. 번역 후 남은 바이트를 맞추려고 의미 없는 문장을 늘리지 않는다.
   자연스럽게 축약한 뒤 기존 관례의 공백·널 또는 검증된 압축 키로
   길이를 맞춘다.

## 생성 절차

1. 원본 `*_jpn.json`을 복사해 `*_kor.json`을 만든다.
2. 원칙적으로 값만 번역한다. 끝 주소 확장이 필요하면 원본 바이너리의
   연속된 `00` 영역과 다음 항목의 시작점을 먼저 검증한다.
3. 기존 `*_kor.json`을 갱신하는 작업이라면 그 파일의 `binary_input`
   같은 프로젝트별 메타데이터를 보존한다.
4. `zero_padding`, `encoding`, `custom_input`, `binary_input`은 일반 대사
   항목이 아니다. 기존 설정이나 명확한 패치 자료 없이 새로 만들거나
   번역하지 않는다.
5. 각 항목마다 주소 길이와 번역문 길이를 계산해 정확히 맞춘다.
6. 동일 원문이 여러 곳에 나오면 기존 번역을 검색해 용어와 말투를
   통일한다.
7. 전체 파일의 주소 집합, 지원 문자, 바이트 길이를 검사한 후 바이너리
   기록을 수행한다.

## 완료 전 검증

- 원본과 번역본의 `=` 포함 주소 키 집합이 완전히 같은가?
- 모든 주소가 `START <= END`인 올바른 16진수 범위인가?
- 모든 번역문의 계산 길이가 해당 주소 길이와 정확히 같은가?
- 모든 `{키}`가 `custom_word.json`에 존재하는가?
- 모든 일반 문자와 `|X` 문자가 각각의 한국어 2바이트/1바이트 폰트
  테이블에 존재하는가?
- 일본어 한자·가나 또는 미번역 문장이 남아 있지 않은가?
- `%s`, `%d` 등의 서식 인자와 필수 제어 코드가 보존되었는가?
- `0x:` 값과 `custom_input`의 원시 바이트 길이가 주소 길이와 같은가?
- JSON 파싱이 성공하는가?
- 워크스페이스 번호와 플랫폼을 올바르게 설정한 뒤
  `write_script_auto.py`를 실행했을 때 주소 차이가 0이고 유효 문장 비율이
  100%인가?

주소 차이, 길이 불일치, 미지원 문자 또는 100% 미만의 기록 결과가 하나라도
있으면 완성본으로 간주하지 않는다.

## 추가 문체 및 표기 규칙

1. 방언 원문도 방언으로 옮기지 않고 일반적인 한국어로 번역한다.
2. 일반 보고, 질문, 안내 문장에는 반말을 사용하지 않는다. 반말은 전투
   중 상대편에게 직접 하는 대사임이 명확한 경우에만 허용한다.
3. 공간에 여유가 있으면 명사형이나 지나친 축약형보다
   `~합니다|.`, `~했습니다|.` 같은 완전한 문장형을 우선한다.
   명사형 UI 문구는 완전한 문장이 주소에 들어가지 않는 짧은 칸에만 쓴다.
4. 실제 문장이 '다'로 끝나면 전각 마침표 `.` 또는 반각 마침표 `|.` 순으로 우선순위를 두어 붙인다. 단, 공간이 부족할 시 반각 마침표를 사용한다. 같은 범위 내 문장에 하나라도 반각 마침표가 있으면 마침표를 모두 반각 마침표로 통일한다. 기존 널 패딩으로 넣을 수 없으면 확인된 빈 공간을 사용해 양쪽 JSON의 끝 주소를 필요한 만큼만 확장한다.
5. 쉼표나 마침표 자체의 글자 폭에 공백이 포함되므로 뒤에 별도 `|_`를 붙이지 않는다. 문맥상 추가 공백이 실제로 필요한 경우만 예외로 한다.
6. `|␂`로 문장이 나뉘는 경우, 그룹 단위의 원래 바이트 길이를 유지하기 위해 최대한 대사를 채우되 필요시 전각 공백(`_`) 또는 반각 공백(`|_`)을 사용해 길이를 유지하려는 노력을 한다. 이는 대사 줄바꿈 코드(`|␂`)를 사용하면서도 원본 문장의 길이를 보존하기 위함이다.
7. 서로 이어 붙여 표시하는 조각 문장은 개별 항목만 직역하지 않는다. 인접 문자열과 실행 코드의 출력 순서를 확인해, 연결했을 때 자연스러운 한국어 어순과 조사가 되도록 각 조각을 번역한다.
8. 성명 조합표, 입력 후보표, 제어용 바이트열처럼 문장이 아닌 데이터는 번역하지 않고 원본 그대로 유지한다.

## 실행 파일의 조각 문장 확인
실행 파일에서 단독으로 불완전한 문장이 발견되면 다음 순서로 확인한다.

1. 같은 파일의 앞뒤 주소에 있는 문자열과 서식 인자를 확인한다.
2. `reverse/`의 대응 ASM에서 문자열을 사용하는 루틴과 출력 순서를
   추적한다.
3. 문자열이 실행 이미지 밖의 런타임 로드 데이터라 ASM에 직접
   라벨링되지 않은 경우, 파일 오프셋·인접 항목·제어 코드·서식 인자의
   연결 관계를 함께 근거로 사용한다.
4. 연결 문장 전체를 기준으로 높임말, 조사, 문장부호와 줄바꿈을 정한다.
